About Multi-Server Configuration

Terminology

A multi-server configuration is composed of a master server and several nodes.

A master server:

Represents an entry point to all administration operations.

Contains the definition of all backup configuration objects.

Centralizes license management.

Assigns a tenant and a node to any new agent.

A node:

Is a standalone server used to contain information regarding the agents and tenants associated with this node. For more information on tenants, see Tenants.

Contains up to 2000 agents. If a problem occurs on a node, only the agents on this node are impacted.

In the , you cannot create a node, but you can modify some properties of an existing node.

Data transfer between nodes, agents and the master server

In a multi-server configuration, only the nodes and agents communicate with the master server. For instance, a agent can communicate with the master server so that the master server assigns the agent to a specific node. The number of nodes that can be created in a multi-server configuration is not limited.

The is used to:

Centralize or edit information on strategies, users and user profiles, protection policies, rules, zones, paths, files categories or agents.

Manage restore and backups on any or all servers known from the master server.

Centralize reporting on agents, tenants, notifications, nodes and the master server. In the , a superadmin can display nodes and tenants associated with an agent (agents view) or notifications sent from nodes (notifications view), using the button below the total number of items.

Limitations

It is not possible to:

Switch from a standalone configuration with existing agents to a multi-server configuration. To implement a multi-server configuration, you must install new servers and agents.

Implement a multi-server configuration in a version below .

Implement a multi-server configuration on .

Use simultaneously a standalone configuration and a multi-server configuration.
